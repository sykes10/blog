## Phase Goal

**As a** visiting engineer or hiring manager, **I want to** read one fully realized, correctly themed, syntax-highlighted, SEO-correct Pattern post, **so that** I can learn a production-grade mental model and judge how the author thinks.

> Note: the ROADMAP `**Goal:**` line is outcome-shaped rather than literal "As a / I want to / so that" form. The user story above is derived faithfully from that goal and the Phase 1 success criteria for MVP-mode planning. It was not invented.

## This Plan's Slice (Walking Skeleton)

This is the Walking Skeleton plan — the thinnest end-to-end stack that proves the whole pipeline works: project scaffold + routing + one real MDX read/render through the typed content layer + one real UI interaction (theme toggle) + a documented local full-stack run command + a smoke test. After this plan, a visitor can load `/patterns/toast-notification-system`, see real rendered MDX content, and toggle light/dark theme with no flash. Syntax highlighting, the full 8-section post body, and SEO/RSS/sitemap are layered on in Wave 2 (Plans 02 and 03).

<objective>
Scaffold a Next.js 16 App Router project (TypeScript + Tailwind v4 + Turbopack) and wire the Velite -> next-mdx-remote/rsc content pipeline end-to-end so a single Pattern post renders at its route, with flash-free light/dark theming and responsive prose typography.

Purpose: Lock the most expensive-to-unwind foundational decisions (content layer wiring under Turbopack, MDX rendering boundary, theming mechanism) by proving them with one real post before any further content is written.
Output: A buildable, runnable site with one Pattern post route, a theme toggle, and a Playwright smoke test that exercises the full stack.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| author MDX file -> build pipeline | The only "input" in this phase is the author's own committed MDX frontmatter + body; Velite's Zod schema is the validation boundary (fails the build on malformed frontmatter). No untrusted external input exists in Phase 1. |
| build output -> browser | Static HTML + a next-themes-injected blocking inline script are delivered to the browser; the inline script contains no interpolated user data. |

| T-01-02 | Information Disclosure | next-themes blocking inline script in `<head>` | low | accept | The injected script is library-controlled with no user-data interpolation; no CSP is configured in Phase 1 (noted for a future phase if a strict CSP is added). |
